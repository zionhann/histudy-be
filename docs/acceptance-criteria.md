# 인수 조건

## 문서 목적

이 문서는 테스트 작성 시 기준으로 삼을 기능별 인수 조건을 정리한다.
여기에는 성공 조건, 실패 조건, 권한 제약, 상태 변화, 응답 수준에서 반드시 확인할 내용을 담는다.

세부 구현보다 외부에서 관찰 가능한 동작을 우선한다.
상태 코드가 현재 코드에서 명확히 정해진 경우에는 그 기준을 따른다.
정책은 분명하지만 구현이 아직 완전히 강제하지 않는 부분은 별도 메모로 남긴다.

## 공통 인수 조건

- 공개 API를 제외한 보호된 API는 유효한 토큰이 없으면 실패해야 한다.
- 토큰이 없거나 유효하지 않으면 `401 Unauthorized`를 반환한다.
- 토큰은 유효하지만 역할이 맞지 않으면 `403 Forbidden`을 반환한다.
- 현재 학기에 의존하는 기능은 현재 학기가 없으면 실패해야 하며, 현재 코드 기준으로 `404 Not Found`를 반환한다.
- 잘못된 요청 형식이나 필수 값 누락은 `400 Bad Request`를 반환한다.
- 중복 생성처럼 충돌 의미가 분명한 경우는 `409 Conflict`를 반환한다.
- 예외 매핑이 없는 런타임 오류는 현재 코드 기준으로 `500 Internal Server Error`가 될 수 있다.
- 마스킹이 적용되는 조회 API는 학번 등 개인 식별 정보가 그대로 노출되지 않아야 한다.
- API 계약이 바뀌면 테스트와 함께 `api-docs.yaml`도 갱신해야 한다.

## 공개 API

### 공개 팀 랭킹 조회

- `/api/public/teams`는 인증 없이 호출할 수 있어야 한다.
- 현재 학기 기준 그룹 목록을 반환해야 한다.
- 랭킹은 총 학습 시간 내림차순으로 정렬되어야 한다.
- 각 팀의 대표 이미지는 최신 리포트 기준으로 계산되어야 한다.
- 현재 학기가 없으면 실패해야 한다.

### 공개 활동 지표 조회

- `/api/public/activity`는 인증 없이 호출할 수 있어야 한다.
- `term=all`이면 전체 기준 지표를 반환해야 한다.
- `term=current`이면 현재 학기 기준 지표를 반환해야 한다.
- 잘못된 `term` 값은 현재 코드 기준으로 `all`과 동일하게 처리된다.
- `term=current`에서 현재 학기가 없으면 실패해야 한다.

### 공개 배너 조회

- `/api/public/banners`는 인증 없이 호출할 수 있어야 한다.
- 활성화된 배너만 반환해야 한다.
- 배너 순서는 `displayOrder` 오름차순이어야 한다.

## 인증과 사용자 등록

### 회원가입

- `/api/users`는 새 사용자를 생성하고 즉시 토큰을 발급해야 한다.
- 성공 시 `isRegistered=true`, `Bearer` 토큰 정보, `USER` 역할을 반환해야 한다.
- 동일한 `sub`로 이미 가입된 사용자는 `409 Conflict`로 실패해야 한다.

### 로그인

- `/api/auth/login`은 `sub`가 있으면 등록된 사용자를 찾아 토큰을 발급해야 한다.
- 관리자는 `ADMIN`, 현재 학기 그룹이 있는 신청자는 `MEMBER`, 그 외는 `USER` 역할로 로그인되어야 한다.
- `sub`가 없으면 `400 Bad Request`로 실패해야 한다.
- 사용자를 찾지 못하면 현재 코드 기준으로 `404 Not Found`와 `isRegistered=false` 응답을 반환한다.

### 액세스 토큰 재발급

- `/api/auth/token`은 유효한 refresh token으로 새 access token을 발급해야 한다.
- refresh token이 없으면 `401 Unauthorized`로 실패해야 한다.
- refresh token이 유효하지 않으면 `401 Unauthorized`로 실패해야 한다.

## 사용자 조회와 내 정보

### 사용자 검색

- `/api/users`와 `/api/v2/users`는 인증된 사용자만 호출할 수 있어야 한다.
- 검색 결과에서 관리자 계정은 제외되어야 한다.
- 검색 결과에서 자기 자신은 제외되어야 한다.
- 검색어가 비어 있으면 전체 사용자 목록을 학번 오름차순으로 반환해야 한다.
- `v2` 응답은 마스킹된 사용자 정보를 반환해야 한다.

### 내 정보 조회

- `/api/users/me`는 인증된 사용자가 자신의 기본 정보를 조회할 수 있어야 한다.
- 현재 코드 기준으로 `USER`, `MEMBER`, `ADMIN` 모두 접근 가능하다.

### 내 신청서 조회

- `/api/users/me/forms`와 `/api/v2/users/me/forms`는 `USER` 권한만 접근 가능하다.
- 현재 학기 신청 내역이 있으면 신청한 친구와 과목 목록을 반환해야 한다.
- 현재 학기 신청 내역이 없으면 빈 배열을 반환해야 한다.
- `v2` 응답은 친구 정보에 마스킹을 적용해야 한다.

## 스터디 신청

### 신청서 등록

- `/api/forms`와 `/api/v2/forms`는 `USER` 권한만 접근 가능하다.
- 신청 시 현재 학기, 친구 목록, 과목 목록이 현재 사용자 기준으로 연결되어 저장되어야 한다.
- 기존 신청서가 있고 아직 그룹에 배정되지 않았다면 새 제출로 대체되어야 한다.
- 상호 친구 신청이 성립하는 경우 관련 요청 상태는 수락 상태로 반영되어야 한다.

### 신청 실패 조건

- 현재 학기가 없으면 실패해야 한다.
- 존재하지 않는 사용자나 과목을 참조하면 실패해야 한다.
- 이미 그룹에 배정된 신청자는 재신청할 수 없어야 한다.
- 이 경우 현재 구현은 비즈니스 예외로 정리되어 있지 않아 `500`이 날 수 있으므로, 테스트에서는 정책과 실제 응답을 분리해 다룰 필요가 있다.

## 과목 관리

### 과목 목록 조회

- `/api/courses`는 `ADMIN`과 `USER`가 호출할 수 있어야 한다.
- 검색어가 없거나 공백이면 현재 과목 전체를 반환해야 한다.
- 검색어가 있으면 trim 후 검색해야 한다.

### 과목 일괄 등록

- `/api/courses` `POST`는 `ADMIN`만 호출할 수 있어야 한다.
- CSV 파일이 비어 있지 않으면 과목 목록을 교체하고 `201 Created`를 반환해야 한다.
- 빈 파일이면 현재 코드 기준 `406 Not Acceptable`로 실패해야 한다.

### 과목 삭제

- `/api/courses/delete`는 `ADMIN`만 호출할 수 있어야 한다.
- 삭제 결과는 현재 코드 기준 정수 값으로 반환된다.

## 팀, 리포트, 이미지

### 팀 리포트 생성과 조회

- `/api/team/reports` 생성과 목록 조회는 `MEMBER` 권한이 필요하다.
- 리포트 생성 시 현재 학기와 현재 사용자의 그룹이 기준이 되어야 한다.
- 목록 조회는 현재 사용자의 그룹 리포트를 최신순으로 반환해야 한다.

### 단건 리포트 조회

- `/api/team/reports/{reportId}`는 `MEMBER`와 `ADMIN`이 조회할 수 있어야 한다.
- 리포트가 없으면 `404 Not Found`를 반환해야 한다.

### 리포트 수정과 삭제

- `/api/team/reports/{reportId}` 수정과 삭제는 `MEMBER` 권한이 필요하다.
- 리포트가 없으면 `404 Not Found`를 반환해야 한다.
- 리포트 수정 시 참여자, 과목, 이미지 목록이 새 요청 기준으로 반영되어야 한다.

### 팀 구성원과 과목 조회

- `/api/team/users`, `/api/team/courses`는 `MEMBER` 권한이 필요하다.
- 팀 구성원 조회는 마스킹된 사용자 정보를 반환해야 한다.
- 팀 과목 조회는 현재 사용자의 현재 학기 그룹 기준으로 계산되어야 한다.

### 리포트 이미지 업로드

- `/api/team/reports/image`와 `/api/team/reports/{reportId}/image`는 `MEMBER` 권한이 필요하다.
- 업로드 성공 시 `imagePath`를 반환해야 한다.
- 기존 리포트 기준 업로드에서 동일 바이너리 이미지가 있으면 기존 경로를 재사용해야 한다.

### 소유권 제약 메모

- 원칙상 리포트 작성과 수정은 자신의 그룹 범위 안에서만 이뤄져야 한다.
- 현재 구현은 역할과 현재 그룹 존재는 검사하지만, 일부 리포트 단건 수정/삭제/조회에서 그룹 소유권 검증이 명시적이지 않다.
- 이 영역의 테스트는 현재 회귀 기준과 목표 정책을 구분해 작성하는 것이 안전하다.

## 관리자 기능

### 그룹 운영 조회

- `/api/admin/manageGroup`, `/api/admin/groupReport/{id}`, `/api/admin/allUsers`, `/api/admin/unmatched-users`, `/api/admin/users/unassigned`는 `ADMIN`만 호출할 수 있어야 한다.
- 각 조회는 현재 학기 기준 데이터를 반환해야 한다.

### 팀 매칭 실행

- `/api/admin/team-match`는 `ADMIN`만 호출할 수 있어야 한다.
- 성공 시 `201 Created`를 반환해야 한다.
- 매칭은 현재 학기의 미배정 신청자만 대상으로 해야 한다.
- 친구 우선 매칭을 먼저 수행하고, 남은 인원에 대해 과목 우선 매칭을 수행해야 한다.
- 그룹 태그는 현재 학기의 최대 태그 다음 번호부터 이어져야 한다.
- 매칭 대상이 없어도 요청 자체는 성공으로 처리된다.

### 학기 관리

- `/api/admin/academicTerm` 생성은 `ADMIN`만 호출할 수 있어야 한다.
- 동일한 연도/학기 조합은 중복 생성할 수 없고 `409 Conflict`로 실패해야 한다.
- `/api/admin/academicTerm` 조회는 학기 목록을 최신 순으로 반환해야 한다.
- `/api/admin/academicTerm/{id}/current`는 지정한 학기를 현재 학기로 만들고, 기존 현재 학기가 있으면 해제해야 한다.
- 존재하지 않는 학기 id는 `404 Not Found`로 실패해야 한다.

### 사용자 신청서 삭제와 사용자 수정

- `/api/admin/form`과 `/api/admin/edit-user`는 `ADMIN`만 호출할 수 있어야 한다.
- 신청서 삭제는 현재 학기 기준으로 동작해야 한다.
- 이미 그룹이 배정된 신청서는 삭제할 수 없어야 한다.
- 사용자 수정은 사용자 기본 정보와 그룹 배정 상태를 함께 반영할 수 있어야 한다.

## 배너 관리

### 관리자 배너 조회

- `/api/admin/banners`는 `ADMIN`만 호출할 수 있어야 한다.
- 전체 배너를 `displayOrder` 오름차순으로 반환해야 한다.

### 배너 생성

- `/api/admin/banners` `POST`는 `ADMIN`만 호출할 수 있어야 한다.
- 이미지와 label은 필수다.
- 이미지는 5MB 이하여야 하며 실제 이미지 파일이어야 한다.
- redirectUrl이 있으면 `http` 또는 `https` URL이어야 한다.
- `active`가 비어 있으면 기본값은 `true`다.
- 성공 시 `201 Created`와 생성된 배너 정보를 반환해야 한다.

### 배너 수정

- `/api/admin/banners/{bannerId}`는 `ADMIN`만 호출할 수 있어야 한다.
- label, redirectUrl, active, image 중 하나 이상이 있어야 한다.
- 수정할 값이 하나도 없으면 `400 Bad Request`로 실패해야 한다.
- 존재하지 않는 배너는 `404 Not Found`로 실패해야 한다.

### 배너 순서 변경

- `/api/admin/banners/reorder`는 `ADMIN`만 호출할 수 있어야 한다.
- 현재 존재하는 모든 배너 id를 정확히 한 번씩 포함한 순서 목록이어야 한다.
- 누락, 중복, 잘못된 id가 있으면 `400 Bad Request`로 실패해야 한다.
- 배너가 하나도 없는 상태에서 빈 순서 목록이면 성공으로 처리된다.

### 배너 삭제

- `/api/admin/banners/{bannerId}` `DELETE`는 `ADMIN`만 호출할 수 있어야 한다.
- 삭제 후 남은 배너의 `displayOrder`는 빈 번호 없이 다시 정렬되어야 한다.
- 존재하지 않는 배너는 `404 Not Found`로 실패해야 한다.
